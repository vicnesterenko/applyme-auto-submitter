# Звіт про виконання тестового завдання (ApplyMe — Automation Engineer)

## 1. Який підхід обрав для проходження Apply на вакансії і чому?
- **Основний підхід:** браузерна автоматизація за допомогою **SeleniumBase** у розширеному UC-режимі (Undetected ChromeDriver).
- **Обґрунтування:** прямі мережеві запити (network requests через API) до Lever швидко блокуються Cloudflare через аналіз TLS-fingerprint (патерни JA3/JA4) та відсутність актуальної антибот-куки `cf_clearance`. UC-режим SeleniumBase нативно маскує WebDriver-флаги, емулює реальні компоненти Chrome, працює через стійкий persistent-профіль і оминає первинні інфраструктурні фільтри Cloudflare, що дозволяє стабільно рендерити та заповнювати форму.

## 2. Чому відхилено Playwright: досвід і виявлені недоліки
На початковому етапі бот реалізовувався на **Playwright (з playwright-stealth)**. Під час живих тестів на кастомних формах Lever (зокрема *Aledade* та *PadSplit*) виявились критичні недоліки:
1. **Слабкий обхід Cloudflare Turnstile / hCaptcha:** стандартне маскування `playwright-stealth` застаріло для сучасних алгоритмів Cloudflare — фронтенд розпізнавав контекст автоматизації та циклічно показував капчу, блокуючи рендеринг форми.
2. **Конфлікт із динамічними React-елементами:** поля Lever загорнуті у складні React-картки. Заповнення через `.fill()` / `.type()` часто спричиняло розсинхронізацію стану React: дані візуально з'являлися в інпуті, але при сабміті форма вважала їх порожніми.
3. **Обмеження CDP у Playwright:** на відміну від SeleniumBase UC, який перехоплює запуск браузера на найнижчому рівні, Playwright використовує власні модифіковані бінарники, що легше детектуються enterprise-системами антибот-захисту (Cloudflare Advanced Bot Management).

Перехід на **SeleniumBase UC** вирішив ці проблеми завдяки нативній симуляції дій користувача (`driver.uc_click`), стійкості до оновлень Chrome та сумісності з кастомними JS-ін'єкціями для обходу React-інпутів.

## 3. Який підхід обрав для проходження капчі і чому?
- **Підхід:** автономний перехоплювач метаданих + обхід через **third-party solver (2Captcha API)**.
- **Обґрунтування:** на Lever використовується hCaptcha, яка активується invisible-челенджами під час взаємодії з динамічними React-картками. Скрипт локалізує елемент `.h-captcha` (або витягує параметри з iframe), бере `data-sitekey` і поточний URL, відправляє їх у 2Captcha та інжектує отриманий токен у приховані елементи `h-captcha-response` і `g-recaptcha-response`, після чого ініціює відправку форми через JS/CDP.

## 4. Які запити робить фронтенд при заповненні форми та подачі заявки?
За аналізом вкладки Network у DevTools:
1. **Парсинг резюме:** при додаванні файлу фронтенд робить асинхронний `POST` на `/parseResume` у форматі `multipart/form-data`; у відповідь повертається JSON із розпарсеними даними (ім'я, пошта, компанії) для автозаповнення.
2. **Динамічна структура карток:** додаткові питання Lever зберігає у прихованих інпутах структури `cards[...][baseTemplate]`. Фронтенд рендерить кастомні React-компоненти з цих JSON-конфігів, де кожне поле має обфускований суфікс (наприклад, `name="cards[ID_картки][field0]"`), що ускладнює роботу простих ботів за стандартними селекторами.
3. **Фінальний сабміт:** подача здійснюється через `POST` на `/apply`, де вся анкета разом із токеном капчі передається єдиним multipart-пакетом.

## 5. Що не вдалось і чому?
За останнім прогоном (`resources/result_report/results.json`) 4 з 5 заявок завершились успішно (`skillerszone`, `theathletic`, `aledade`, `raptv` — статус `success`). Заявка на `padsplit` повернула `failed: captcha_or_timeout`: після сабміту сторінка не редіректнула на підтвердження "Thanks" у відведений таймаут — найімовірніше через invisible-челендж hCaptcha або повільний рендер кастомних карток. Стан зафіксовано у звіті та на скріншоті `error_sbase_*`.

## 6. Що треба доопрацювати для production-рівня та скейлу на Х1000 аплаїв?
1. **Проксі-ротація:** інтеграція резидентних/мобільних проксі зі sticky-сесіями, щоб Lever не блокував IP-пул сервера при масових запитах. Можемо спробувти використати цього провайдера https://dashboard.decodo.com/welcome.
2. **Паралелізація та Docker-пули:** перехід від лінійного циклу до асинхронної черги задач - RabbitMQ.
3. **ШІ-мапінг кастомних питань (LLM routing):** заміна жорстких регулярних виразів та текстових закладок на ШІ-агента. Текст кожного кастомного запитання надсилається на швидку LLM - (GPT-4o-mini, Qwen), яка на основі профілю кандидата генерує точну відповідь під конкретне поле.
4. **No-Code/Low-Code оркестрація (n8n):** щоб не перевантажувати бекенд кодом підтримки обходу ботів, життєвий цикл заявки (ротація токенів, логування, моніторинг станів 2Captcha) доцільно винести на рівень оркестрації в [n8n](https://n8n.io/workflows/7151-bypass-cloudflare-turnstile-for-web-scraping-with-2captcha/).

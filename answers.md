# Звіт про виконання тестового завдання

## 1. Який підхід обрав для проходження Apply на вакансії і чому?
- **Основний підхід:** Браузерна автоматизація за допомогою **SeleniumBase** у розширеному UC-режимі (Undetected ChromeDriver) - https://www.youtube.com/watch?v=aF3ee0uTBm4
- **Обґрунтування:** Прямі мережеві запити (network requests через API) до Lever миттєво блокуються Cloudflare через аналіз TLS-fingerprint (патерни JA3/JA4) та відсутність актуальної антибот-куки `cf_clearance`. UC-режим SeleniumBase нативно маскує WebDriver-флаги, емулює реальні компоненти Chrome, працює через стійкий persistent-профіль і оминає первинні інфраструктурні фільтри Cloudflare, що дозволяє стабільно рендерити та заповнювати форми.
- *Примітка щодо Headless:* Використання `headless=True` разом із UC-режимом знижує рівень маскування (stealth). Для production-середовищ без GUI (наприклад, Linux-сервери) оптимумом є використання утиліти **Xvfb**, яка дозволяє запускати повноцінний headed-браузер у віртуальному вікні.

## 2. Чому відхилено Playwright: досвід і виявлені недоліки
На початковому етапі бот реалізовувався на **Playwright (з бібліотекою playwright-stealth)**. Під час тестування на кастомних формах Lever (зокрема *Aledade* та *PadSplit*) виявилися критичні недоліки:
1. **Застарілі алгоритми маскування:** Стандартні сигнатури `playwright-stealth` уже розпізнаються сучасними алгоритмами Cloudflare Advanced Bot Management — фронтенд детектував контекст автоматизації, циклічно викликаючи капчу та блокуючи рендеринг професійних форм.
2. **Обмеження модифікованих бінарників:** Playwright використовує власні модифіковані збірки браузерів. На відміну від SeleniumBase UC, який взаємодіє з нативним Chrome та перехоплює процеси на низькому рівні, сигнатури Playwright легше піддаються enterprise-детекту.

Переход на **SeleniumBase UC** вирішив ці проблеми завдяки нативній симуляції дій користувача (`driver.uc_click`), стійкості до оновлень Chrome та коректній роботі з кастомними JS-ін'єкціями для тригерингу подій у React-інпутах.

## 3. Який підхід обрав для проходження капчі і чому?
- **Підхід:** Автономний перехоплювач метаданих + обхід через **third-party solver (2Captcha API)** - https://2captcha.com/enterpage
- **Обґрунтування:** На платформах типу Lever використовується hCaptcha, яка активується invisible-челенджами під час взаємодії з динамічними React-картками. Скрипт локалізує елемент `.h-captcha` (або витягує параметри з iframe), зчитує `data-sitekey` та поточний URL сторінки, після чого відправляє таску в 2Captcha. Отриманий токен динамічно інжектується через JavaScript у приховані елементи `h-captcha-response` і `g-recaptcha-response`, що дозволяє успішно виконати сабміт форми.

## 4. Які запити робить фронтенд при заповненні форми та подачі заявки?
За результатами аналізу вкладки Network у DevTools:
1. **Парсинг резюме:** При завантаженні файлу фронтенд робить асинхронний `POST`-запит на ендпоінт `/parseResume` у форматі `multipart/form-data`. У відповідь повертається JSON із розпарсеними сутностями (персональні дані, контакти, досвід) для первинного автозаповнення.

3. **Динамічна структура карток:** Додаткові (кастомні) питання Lever зберігає у структурах виду `cards[...][baseTemplate]`. Фронтенд рендерить кастомні React-компоненти з цих JSON-конфігів, де кожне поле отримує обфускований суфікс (наприклад, `name="cards[ID_картки][field0]"`). Це унеможливлює використання статичних hardcoded-селекторів.
4. **Фінальний сабміт:** Подача анкети здійснюється через `POST`-запит на `/apply`, де вся структура форми разом із валідованим токеном капчі передається єдиним multipart-пакетом.

## 5. Що не вдалось і чому?
Згідно з результуючим файлом `resources/result_report/results.json`, 4 з 5 заявок були успішно відправлені (`skillerszone`, `theathletic`, `aledade`, `raptv` — статус `success`). 

Заявка на `padsplit` повернула статус `failed: captcha_or_timeout`. Після кліку на Submit сторінка не перенаправила користувача на confirm-page "Thanks" у межах встановленого таймауту. Аналіз логів показав, що проблема виникла на етапі передачі токена розв'язання капчі.
## 6. Що треба доопрацювати для production-рівня та скейлу на Х1000 аплаїв?
1. **Ротація резидентних проксі:** Інтеграція провайдерів резидентних або мобільних проксі (наприклад, Decodo — https://dashboard.decodo.com/welcome) з підтримкою sticky-сесій. Це дозволить кожній заявці мати унікальний чистий IP-адрес і захистить пул від масових блокувань з боку Cloudflare.
2. **Розподілена архітектура черг (Паралелізація):** Перехід від синхронного лінійного циклу до черг задач на базі **RabbitMQ / Celery**. Це дозволить запускати ізольовані воркери в Docker, гнучко керувати пулом одночасних сесій браузерів (concurrency limits) та реалізувати логіку автоматичних ретраїв (retries) при таймаутах.
3. **ШІ-мапінг кастомних питань (LLM Routing):** Повна заміна жорстких регулярних виразів та текстових евристик на інтеграцію з легкими LLM (наприклад, GPT-4o-mini/Gemini Flash/Claude). Текст лейблів та конфігурація полів зчитуються з DOM-дерева сторінки та передаються в LLM разом із JSON-профілем кандидата. Модель через Structured Outputs повертає готовий мапінг відповідей для заповнення.
4. **No-Code/Low-Code оркестрація (n8n):** Винесення життєвого циклу заявки, моніторингу станів токенів 2Captcha, логування результатів та бізнес-метрики на рівень оркестратора **n8n** (https://n8n.io/workflows/7151-bypass-cloudflare-turnstile-for-web-scraping-with-2captcha/). Це спростить підтримку логіки обходу ботів, зменшить завантаження основного бекенду та надасть зручний дашборд для контролю витрат балансу капчі і моніторингу помилок.

# ApplyMe Auto Submitter

Submit job applications to [Lever](https://www.lever.co/) on behalf of a
candidate **through Lever's official, public [Postings API](https://github.com/lever/postings-api)** —
not via browser automation, CAPTCHA solving, or anti-bot evasion.

The service takes a candidate profile, a resume, and a list of Lever posting
URLs, then validates each posting and (when a Postings API key is configured)
submits an application, returning a structured per-posting result.

---

## How it works

For every posting URL the service:

1. **Parses** the URL into `site` + `posting_id` (handles the `/apply` suffix,
   query strings, fragments, and the EU instance `jobs.eu.lever.co`).
2. **Fetches** the posting JSON (`GET /v0/postings/{site}/{id}?mode=json`) to
   confirm it is published and read its title.
3. In **`dry_run`** mode, stops there and records status `dry_run`.
4. Otherwise looks up the site's Postings API key and:
   - **submits** the application via `POST /v0/postings/{site}/{id}?key={api_key}`
     (multipart form with the resume PDF) and records `success` with an
     `applicationId`;
   - records `failed:<detail>` on error;
   - records `skipped:no_api_key` when no key is configured for that site.

A randomised politeness delay separates consecutive submissions, and transient
HTTP errors (`429/500/502/503/504`, network errors) are retried with
exponential backoff + jitter.

> The apply endpoint requires a Postings API key that the **site owner**
> generates in Lever (Settings → Integrations and API → API Credentials →
> Postings API) and shares with the integration. This is by design — it is the
> sanctioned way to submit applications programmatically. Requests use an honest
> `User-Agent`; no stealth or detection evasion is performed.

## Layout

```
applyme/
  api.py          FastAPI app (routes, lifespan-managed shared HTTP client)
  service.py      orchestration: apply to a list of postings, record results
  lever.py        official Lever Postings API client (httpx) + URL parsing
  profile.py      profile/resume parsing + resume-PDF generation
  models.py       Pydantic models (Candidate, JobPosting, results)
  config.py       env-based settings (Postings API keys, timeouts, delays)
  cli.py          `python -m applyme.cli` runner -> results.json
  templates.py    minimal HTML upload form
  sample_data.py  bundled sample profile/resume + 5 test posting URLs
tests/            pytest suite (profile, lever client, API)
```

## Setup

```bash
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt           # runtime
pip install -r requirements-dev.txt        # + tests/lint
cp .env.example .env                       # add Postings API key(s) if you have them
```

## Run

```bash
uvicorn app:app --reload                   # http://127.0.0.1:8000  (Swagger docs at /docs)
python -m applyme                          # same server
python -m applyme.cli                      # CLI dry-run over the bundled sample -> results.json
```

## Test & lint

```bash
pytest
ruff check .
```

## HTTP API

| Method | Path      | Description                                                  |
| ------ | --------- | ------------------------------------------------------------ |
| `GET`  | `/`       | Minimal HTML upload form (resume + profile + URLs)           |
| `GET`  | `/health` | Liveness probe — `{"status": "ok", "version": "..."}`        |
| `POST` | `/apply`  | Upload resume + profile and submit/validate applications     |
| `GET`  | `/docs`   | Interactive Swagger UI (auto-generated by FastAPI)           |

### `POST /apply`

`multipart/form-data` request:

| Field     | Type     | Required | Description                                                       |
| --------- | -------- | -------- | ----------------------------------------------------------------- |
| `resume`  | file     | yes      | Resume — `.pdf` (passed through) or `.docx/.txt/.md` (rendered to PDF) |
| `profile` | file     | yes      | Candidate profile — `.json`, or `.docx` with embedded profile JSON |
| `urls`    | string   | no       | Lever posting URLs separated by newline, comma, or space. Falls back to the bundled sample postings if empty |
| `dry_run` | boolean  | no       | `true` (default) = validate only, do not submit                   |

Example:

```bash
curl -X POST http://127.0.0.1:8000/apply \
  -F resume=@resume.pdf \
  -F profile=@profile.json \
  -F 'urls=https://jobs.lever.co/raptv/57dfc4b3-7853-401b-9887-459b23a58457' \
  -F dry_run=true
```

Response (`BatchSummary`):

```json
{
  "candidate_name": "Jane Doe",
  "candidate_email": "jane@example.com",
  "dry_run": true,
  "total": 1,
  "succeeded": 1,
  "results": [
    {
      "url": "https://jobs.lever.co/raptv/57dfc4b3-7853-401b-9887-459b23a58457",
      "site": "raptv",
      "posting_id": "57dfc4b3-7853-401b-9887-459b23a58457",
      "title": "Social Editor (Pop Hive)",
      "status": "dry_run",
      "application_id": null
    }
  ]
}
```

`422` is returned when the profile cannot be parsed or is missing required
fields (name / email).

### Result statuses

| Status                | Meaning                                                        |
| --------------------- | -------------------------------------------------------------- |
| `success`             | Application submitted; `application_id` is set                 |
| `dry_run`             | Posting validated, nothing submitted (dry-run mode)            |
| `skipped:no_api_key`  | No Postings API key configured for that site                  |
| `failed:<detail>`     | Submission or validation failed (`<detail>` explains why)      |

## CLI

```bash
# Dry-run (default) against the bundled sample profile + 5 test postings:
python -m applyme.cli

# Real submission (requires APPLYME_LEVER_API_KEY / per-site keys):
python -m applyme.cli --no-dry-run --profile profile.json --resume resume.pdf \
  https://jobs.lever.co/<site>/<posting-id>
```

| Argument             | Description                                          |
| -------------------- | ---------------------------------------------------- |
| `urls...`            | Posting URLs (positional; defaults to bundled sample) |
| `--profile PATH`     | Profile file (`.json`/`.docx`)                       |
| `--resume PATH`      | Resume file (`.pdf`/`.docx`/`.txt`/`.md`)            |
| `--dry-run` / `--no-dry-run` | Validate only vs. actually submit (default: dry-run) |
| `--out PATH`         | Where to write results JSON (default `results.json`) |

Results are written to `results.json` and printed as a status table.

## Configuration (env, prefix `APPLYME_`)

All values are optional with sane defaults; copy `.env.example` to `.env`.
Secrets are never hard-coded — keys are read from the environment so they can be
injected by a secret manager in production.

| Variable                  | Default | Description                                                   |
| ------------------------- | ------- | ------------------------------------------------------------- |
| `APPLYME_LEVER_API_KEY`   | —       | Default Postings API key                                      |
| `APPLYME_LEVER_API_KEYS`  | `{}`    | JSON map `{"<site>": "<key>"}` for per-site keys (overrides default) |
| `APPLYME_SILENT`          | `false` | Suppress Lever's confirmation email to the candidate          |
| `APPLYME_REQUEST_TIMEOUT` | `30`    | HTTP request timeout (seconds)                                |
| `APPLYME_MAX_RETRIES`     | `3`     | Retry attempts for transient HTTP/network errors              |
| `APPLYME_MIN_DELAY`       | `1`     | Min politeness delay between submissions (seconds)            |
| `APPLYME_MAX_DELAY`       | `3`     | Max politeness delay between submissions (seconds)            |
